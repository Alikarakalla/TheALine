# The A Line

A premium fashion-bag e-commerce experience — an animated React storefront plus a
full admin panel, backed by a PHP + MySQL JSON API. **One project, one deploy.**

## Tech stack

- **Frontend:** React 18, Vite, TypeScript, framer-motion, React Router
- **Backend:** PHP 8 + MySQL/MariaDB JSON API (in [`api/`](api/))

## How it's organised (one folder, like a Laravel app)

```
.                 the whole project — upload this folder to Hostinger's public_html
├─ .htaccess      routes the domain: /api -> api/, everything else -> public/
├─ .env           ONE config file (frontend build vars + backend runtime vars)
├─ public/        WEB ROOT — the built storefront (generated by `npm run build`)
├─ api/           PHP backend, served at /api
│  ├─ index.php   front controller
│  ├─ config.php  reads settings from the root .env
│  ├─ core/  controllers/
│  ├─ uploads/    uploaded product images
│  └─ sql/
│     ├─ schema.sql   all 37 tables (structure only)
│     └─ seed.php     creates the admin + baseline settings
├─ static/        files copied verbatim into the build (e.g. the SPA .htaccess)
├─ src/           React source
└─ vite.config.ts
```

The storefront and API are the **same origin**: the app calls `/api/...`, which is
served by the `api/` folder in production and proxied to your local PHP in dev. So
`VITE_API_BASE=/api` works everywhere and you never hard-code a domain.

## Local development

```bash
npm install
npm run dev      # Vite dev server; /api is proxied to your local PHP backend
npm run build    # production build -> public/
```

`npm run dev` proxies `/api` to `http://localhost/lovebag/api` (your XAMPP backend —
see `vite.config.ts`). With no `.env`, the backend also falls back to the local XAMPP
database, so dev usually needs no configuration.

First-time local DB setup: create database `lovebag_db`, import
[`api/sql/schema.sql`](api/sql/schema.sql), then `php api/sql/seed.php`
(creates `admin@lovebag.com` / `admin123`).

## Configuration — one `.env`

Copy `.env.example` to `.env`. The frontend reads `VITE_*` at build time; the PHP
backend reads the rest at runtime from the **same file**.

| Variable | Used by | Example |
|---|---|---|
| `VITE_API_BASE` | frontend (build) | `/api` (default — same origin) |
| `DB_HOST` `DB_PORT` `DB_NAME` `DB_USER` `DB_PASS` | backend | your MySQL credentials |
| `JWT_SECRET` | backend | a long random string |
| `UPLOAD_URL` | backend | `https://your-domain.com/api/uploads` |
| `CORS_ORIGINS` | backend | blank (same domain) |

## Deploy from the terminal (build on your PC, push over SSH)

No File Manager, no FTP. Your computer builds the storefront, then one command ships
`public/`, `api/`, and the root `.htaccess` to Hostinger over SSH. Your `.env` and your
uploaded images stay on the server untouched.

### One-time setup

**1. Enable SSH** — hPanel → Advanced → SSH Access. Note the **IP, username and port**
(usually `65002`). (Recommended: add an SSH key there so deploys are passwordless —
`ssh-keygen -t ed25519`, then paste `~/.ssh/id_ed25519.pub` into hPanel.)

**2. Configure the deploy** — copy the template and fill it in:

```bash
cp scripts/deploy.env.example scripts/deploy.env   # then edit it (gitignored)
```

**3. Create the database** — hPanel → Databases → MySQL. Note the DB name, user, password.

**4. First deploy** (uploads the files, including `api/sql/schema.sql`):

```bash
npm run deploy        # Windows: runs scripts/deploy.ps1 (PowerShell)
                      # macOS/Linux/Git Bash: use  npm run deploy:sh
```

You'll be asked for your SSH password (twice, unless you set up a key).

**5. Set up the server** — SSH in once and finish the install:

```bash
ssh -p 65002 u123456789@123.45.67.89
cd public_html

# a) create the ONE .env at the web root
cat > .env <<'EOF'
VITE_API_BASE=/api
DB_HOST=localhost
DB_NAME=your_db
DB_USER=your_db_user
DB_PASS=your_db_password
JWT_SECRET=paste-a-long-random-string-here
UPLOAD_URL=https://your-domain.com/api/uploads
EOF

# b) load the table structure, then bootstrap the admin + settings
mysql -u your_db_user -p your_db < api/sql/schema.sql
php api/sql/seed.php       # creates admin@lovebag.com / admin123
```

Generate the secret with: `php -r "echo bin2hex(random_bytes(32));"`

**6. Verify** — visit `https://your-domain.com/api/health` (returns `status: up`), then
log into `https://your-domain.com/admin` and **change the admin password immediately**.

### Every deploy after that

```bash
npm run deploy
```

It rebuilds and re-syncs `public/` + `api/` over SSH. The server `.env` and
`api/uploads/` are never overwritten.

### Notes

- Needs Apache `mod_rewrite` (on by default on Hostinger). If routes 404, confirm the
  three `.htaccess` files (root, `public/`, `api/`) uploaded.
- **Uploaded images** are stored in `api/uploads/` (not in git). Image URLs recorded
  locally point at `localhost`; re-upload images through the admin panel on the live
  site, or migrate them separately.
- The `.htaccess` re-exposes the `Authorization` header so admin login works on
  Hostinger's FastCGI PHP.
- The DB committed here is **structure only** (`api/sql/schema.sql`) — no data — so no
  customer/admin records live in this public repo.
